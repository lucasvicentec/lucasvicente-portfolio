1. JavaScript Esencial

JavaScript es la base de todo tu stack. Te van a preguntar sobre esto en el 95% de las entrevistas frontend/fullstack.

Pregunta tipica: "Cual es la diferencia entre let y const?" — const no se puede reasignar, pero si es un objeto/array, sus propiedades SI se pueden mutar.

user.name = "Pedro";  // OK, mutas la propiedad

1.2 Tipos de datos

  • Primitivos: string, number, boolean, null, undefined, symbol, bigint
  • Referencia: object, array, function
Pregunta: "Que diferencia hay entre null y undefined?"
undefined = variable declarada pero sin valor asignado.
null = valor asignado intencionalmente para indicar "vacio".

1.3 Funciones

// Funcion clasica
function sumar(a, b) {
  return a + b;
}

// Arrow function (la que mas usaras en React)
const sumar = (a, b) => a + b;

// Arrow con cuerpo
const procesar = (data) => {
  const resultado = data.filter(x => x > 0);
  return resultado;
};
Diferencia clave: Las arrow functions NO tienen su propio this. Heredan el this del contexto donde se definen. Las funciones clasicas SI tienen su propio this.

1.4 Arrays - Metodos que DEBES saber

Estos los vas a usar todos los dias en React:

const nums = [1, 2, 3, 4, 5];

// MAP - transforma cada elemento, devuelve nuevo array
nums.map(n => n * 2);           // [2, 4, 6, 8, 10]

// FILTER - filtra elementos, devuelve nuevo array
nums.filter(n => n > 3);        // [4, 5]

// FIND - devuelve el PRIMER elemento que cumple
nums.find(n => n > 3);          // 4

// REDUCE - acumula valores
nums.reduce((acc, n) => acc + n, 0);  // 15

// SOME - al menos uno cumple? (boolean)
nums.some(n => n > 4);          // true

// EVERY - todos cumplen? (boolean)
nums.every(n => n > 0);         // true

// INCLUDES - contiene el valor?
nums.includes(3);               // true

// forEach - ejecuta funcion por cada elemento (NO devuelve nada)
nums.forEach(n => console.log(n));
CLAVE: map, filter y reduce NO mutan el array original. Devuelven uno nuevo. Esto es fundamental en React donde la inmutabilidad importa.

1.5 Destructuring

// Objetos
const user = { name: "Lucas", age: 25, city: "Madrid" };
const { name, age } = user;     // name = "Lucas", age = 25

// Arrays
const [first, second] = [10, 20];  // first = 10, second = 20

// Con valores por defecto
const { role = "dev" } = user;  // role = "dev" (no existia)

// Renombrar
const { name: nombre } = user;  // nombre = "Lucas"

1.6 Spread y Rest

// SPREAD - expandir
const arr1 = [1, 2];
const arr2 = [...arr1, 3, 4];   // [1, 2, 3, 4]

const obj1 = { a: 1 };
const obj2 = { ...obj1, b: 2 }; // { a: 1, b: 2 }

// REST - agrupar el "resto"
const { name, ...rest } = user; // rest = { age: 25, city: "Madrid" }

function sum(...nums) {          // nums = [1, 2, 3]
  return nums.reduce((a, b) => a + b, 0);
}

1.7 Promesas y Async/Await

Esto es CRITICO. Toda comunicacion con APIs, bases de datos, etc. es asincrona.

// Promesa basica
fetch("https://api.ejemplo.com/users")
  .then(response => response.json())
  .then(data => console.log(data))
  .catch(error => console.error(error));

// Async/Await (la forma moderna - usa esta)
async function getUsers() {
  try {
    const response = await fetch("https://api.ejemplo.com/users");
    const data = await response.json();
    return data;
  } catch (error) {
    console.error("Error:", error);
  }
}
Pregunta tipica: "Que es una promesa?"
Un objeto que representa un valor que puede estar disponible ahora, en el futuro, o nunca. Tiene 3 estados: pending (esperando), fulfilled (resuelta con valor), rejected (rechazada con error).

1.8 Operador Ternario y Optional Chaining

// Ternario (lo usas MUCHO en JSX)
const msg = age >= 18 ? "Mayor" : "Menor";

// Optional chaining - evita errores si algo es null/undefined
const city = user?.address?.city;  // undefined si algo no existe (no explota)

// Nullish coalescing - valor por defecto solo si null/undefined
const name = user.name ?? "Anonimo";

1.9 Closures

function crearContador() {
  let count = 0;              // esta variable "vive" dentro del closure
  return function() {
    count++;
    return count;
  };
}

const contador = crearContador();
contador(); // 1
contador(); // 2
contador(); // 3
Pregunta: "Que es un closure?"
Una funcion que recuerda las variables del scope donde fue creada, incluso cuando se ejecuta fuera de ese scope. Es como si la funcion "capturara" su entorno.

1.10 Event Loop

Pregunta: "Como funciona el Event Loop en JavaScript?"
JS es single-threaded. El Event Loop es el mecanismo que permite ejecutar codigo asincrono sin bloquear el hilo principal:

1. Call Stack - donde se ejecuta el codigo sincrono
2. Web APIs - donde se procesan las operaciones async (fetch, setTimeout, etc.)
3. Callback Queue - cola de funciones listas para ejecutarse
4. Event Loop - revisa si el Call Stack esta vacio; si lo esta, mueve callbacks de la queue al stack

Por eso un setTimeout(fn, 0) no se ejecuta inmediatamente: primero va a la queue y espera a que el stack se vacie.

2.2 Interfaces vs Types

// Interface - para objetos (preferida para definir "formas")
interface User {
  id: number;
  name: string;
  email: string;
  role?: string;        // ? = opcional
  readonly createdAt: Date;  // no se puede modificar
}

// Type - mas flexible (unions, intersections, primitivos)
type Status = "active" | "inactive" | "banned";  // union literal
type ID = string | number;

// Extender
interface Admin extends User {
  permissions: string[];
}

// Intersection con types
type AdminUser = User & { permissions: string[] };
Pregunta: "Cuando usas interface y cuando type?"
Interface: para definir la forma de objetos y clases. Se pueden extender y mergear.
Type: para unions, intersections, tipos primitivos con alias, tipos condicionales. Mas flexible pero no se puede mergear.

2.3 Generics

// Funcion generica - funciona con cualquier tipo
function firstElement<T>(arr: T[]): T | undefined {
  return arr[0];
}

firstElement<number>([1, 2, 3]);     // 1 (tipo number)
firstElement<string>(["a", "b"]);    // "a" (tipo string)

// Interface generica
interface ApiResponse<T> {
  data: T;
  status: number;
  message: string;
}

// Uso
const response: ApiResponse<User[]> = {
  data: [{ id: 1, name: "Lucas", email: "...", createdAt: new Date() }],
  status: 200,
  message: "OK"
};
Explicacion simple: Los generics son como "huecos" que rellenas con un tipo cuando los usas. Es como decir "esta funcion funciona con CUALQUIER tipo, tu me dices cual cuando la llames".

2.4 Utility Types (los mas comunes)

// Partial - hace todas las propiedades opcionales
type UpdateUser = Partial<User>;  // todo opcional

// Pick - selecciona solo algunas propiedades
type UserPreview = Pick<User, "name" | "email">;

// Omit - excluye propiedades
type UserWithoutId = Omit<User, "id">;

// Record - crea un tipo de objeto con claves y valores tipados
type Roles = Record<string, string[]>;  // { [key: string]: string[] }

Pregunta: "Que es React y por que se usa?"

3.4 Hooks esenciales

useState - manejar estado

import { useState } from "react";

function Counter() {
  const [count, setCount] = useState(0);  // valor inicial: 0

  return (
    <div>
      <p>Count: {count}</p>
      <button onClick={() => setCount(count + 1)}>+1</button>
      <button onClick={() => setCount(prev => prev - 1)}>-1</button>
    </div>
  );
}
Clave: Usa la forma prev => prev + 1 cuando el nuevo estado depende del anterior. Asi evitas bugs por batching.

useEffect - efectos secundarios

import { useEffect, useState } from "react";

function UserList() {
  const [users, setUsers] = useState([]);

  // Se ejecuta al montar el componente (array vacio = solo una vez)
  useEffect(() => {
    async function fetchUsers() {
      const res = await fetch("/api/users");
      const data = await res.json();
      setUsers(data);
    }
    fetchUsers();
  }, []);  // [] = solo al montar

  // Se ejecuta cuando cambia userId
  useEffect(() => {
    console.log("userId cambio");
  }, [userId]);  // se re-ejecuta si userId cambia

  // Cleanup (limpiar al desmontar)
  useEffect(() => {
    const interval = setInterval(() => console.log("tick"), 1000);
    return () => clearInterval(interval);  // cleanup
  }, []);
}
Pregunta: "Que es el array de dependencias de useEffect?"
Es el segundo argumento. Le dice a React CUANDO re-ejecutar el efecto:
- [] = solo al montar (una vez)
- [var1, var2] = cuando var1 o var2 cambien
- Sin array = en CADA render (evitar esto)

useRef - referencia que persiste entre renders

const inputRef = useRef<HTMLInputElement>(null);

// Acceder al DOM directamente
inputRef.current?.focus();

// Tambien para guardar valores sin causar re-render
const renderCount = useRef(0);
renderCount.current++;  // no causa re-render

useMemo y useCallback - optimizacion

// useMemo - memoriza un VALOR calculado
const filteredUsers = useMemo(() => {
  return users.filter(u => u.active);
}, [users]);  // solo recalcula si users cambia

// useCallback - memoriza una FUNCION
const handleClick = useCallback(() => {
  console.log("click");
}, []);  // misma referencia entre renders
No abuses: useMemo y useCallback son optimizaciones. No los uses en todo. Solo cuando tienes calculos costosos o pasas funciones como props a componentes memoizados.

3.6 Listas y Keys

{users.map(user => (
  <UserCard key={user.id} name={user.name} />
))}
Pregunta: "Por que se necesita key en las listas?"
React usa las keys para identificar que elementos cambiaron, se agregaron o se eliminaron. Sin keys unicas, React no puede optimizar el re-renderizado y puede causar bugs visuales. NUNCA uses el indice del array como key si la lista puede cambiar de orden.

3.7 Manejo de formularios

function LoginForm() {
  const [email, setEmail] = useState("");
  const [password, setPassword] = useState("");

  const handleSubmit = (e: React.FormEvent) => {
    e.preventDefault();  // evita recarga de pagina
    console.log({ email, password });
  };

  return (
    <form onSubmit={handleSubmit}>
      <input
        type="email"
        value={email}
        onChange={(e) => setEmail(e.target.value)}
        placeholder="Email"
      />
      <input
        type="password"
        value={password}
        onChange={(e) => setPassword(e.target.value)}
      />
      <button type="submit">Login</button>
    </form>
  );
}

3.8 Context API

// 1. Crear contexto
const ThemeContext = createContext<"light" | "dark">("light");

// 2. Provider (envuelve componentes que necesitan acceso)
function App() {
  const [theme, setTheme] = useState<"light" | "dark">("light");
  return (
    <ThemeContext.Provider value={theme}>
      <Page />
    </ThemeContext.Provider>
  );
}

// 3. Consumir con useContext
function Button() {
  const theme = useContext(ThemeContext);
  return <button className={theme}>Click</button>;
}
Cuando usar Context: Para datos "globales" como tema, usuario logueado, idioma. NO para estado complejo ni para datos que cambian muy frecuentemente (usa Zustand o Redux para eso).

4. Next.js

Next.js es el framework que usas sobre React. Anade routing, SSR, API routes y mas.

4.1 Que es Next.js

Pregunta: "Que ventajas tiene Next.js sobre React puro?"
- File-based routing: las carpetas y archivos definen las rutas, no necesitas react-router
- Server-Side Rendering (SSR): la pagina se renderiza en el servidor, mejor SEO y carga inicial
- API Routes: puedes crear endpoints backend dentro del mismo proyecto
- Optimizaciones: imagenes, fonts, code splitting automatico
- Server Components (App Router): componentes que se ejecutan en el servidor y no envian JS al cliente

4.2 App Router vs Pages Router

Pages Router (legacy)App Router (actual)
Carpeta /pagesCarpeta /app
getServerSidePropsServer Components por defecto
getStaticPropsfetch con cache options
Todo es Client ComponentServer Components + "use client"

4.3 Server Components vs Client Components

// Server Component (por defecto en App Router)
// Se ejecuta en el servidor, NO envia JS al navegador
async function UserList() {
  const users = await db.query("SELECT * FROM users");  // acceso directo a DB
  return <ul>{users.map(u => <li key={u.id}>{u.name}</li>)}</ul>;
}

// Client Component (necesita interactividad)
"use client";  // esta directiva lo hace client component
import { useState } from "react";

function Counter() {
  const [count, setCount] = useState(0);
  return <button onClick={() => setCount(count + 1)}>{count}</button>;
}
Regla: Usa Server Components por defecto. Solo anade "use client" cuando necesites: useState, useEffect, onClick, onChange, o cualquier interactividad del navegador.

  const users = await db.query("SELECT * FROM users");
  return NextResponse.json(users);
}

4.6 SSR, SSG, ISR

Pregunta: "Diferencia entre SSR, SSG e ISR?"
- SSR (Server-Side Rendering): la pagina se genera en cada peticion. Datos siempre frescos pero mas lento.
- SSG (Static Site Generation): la pagina se genera en build time. Muy rapido pero datos estaticos.
- ISR (Incremental Static Regeneration): SSG pero se regenera cada X segundos. Lo mejor de ambos mundos.

6. Node.js y APIs REST

6.1 Que es Node.js

Pregunta: "Que es Node.js?"
Es un entorno de ejecucion de JavaScript fuera del navegador, construido sobre el motor V8 de Chrome. Permite usar JS en el servidor. Es single-threaded pero usa un modelo de I/O no bloqueante (event loop) para manejar muchas conexiones simultaneas.

6.5 CORS

Pregunta: "Que es CORS?"
Cross-Origin Resource Sharing. Es una politica de seguridad del navegador que bloquea peticiones HTTP a un dominio diferente al de la pagina. Si tu frontend esta en localhost:3000 y tu API en localhost:8000, el navegador bloquea la peticion a menos que el servidor envie las cabeceras CORS correctas (Access-Control-Allow-Origin).

7. SQL y PostgreSQL

Lo usas en casi todos tus proyectos. Saber SQL basico es obligatorio.

7.1 Queries basicas

-- Seleccionar todo
SELECT * FROM users;

-- Seleccionar columnas especificas con condicion
SELECT name, email FROM users WHERE active = true;

-- Ordenar
SELECT * FROM users ORDER BY created_at DESC;

-- Limitar resultados
SELECT * FROM users LIMIT 10 OFFSET 20;  -- pagina 3 de 10 items

-- Contar
SELECT COUNT(*) FROM users WHERE role = 'admin';

-- Insertar
INSERT INTO users (name, email, role)
VALUES ('Lucas', 'contacto@lucasvicente.es', 'dev');

-- Actualizar
UPDATE users SET role = 'admin' WHERE id = 1;

-- Eliminar
DELETE FROM users WHERE id = 1;

7.2 JOINs

-- INNER JOIN: solo filas que coinciden en AMBAS tablas
SELECT u.name, o.total
FROM users u
INNER JOIN orders o ON u.id = o.user_id;

-- LEFT JOIN: todas las filas de la izquierda + coincidencias de la derecha
SELECT u.name, o.total
FROM users u
LEFT JOIN orders o ON u.id = o.user_id;
-- Usuarios SIN pedidos tambien aparecen (con o.total = NULL)
Pregunta: "Diferencia entre INNER JOIN y LEFT JOIN?"
INNER JOIN: solo devuelve filas donde hay coincidencia en ambas tablas.
LEFT JOIN: devuelve TODAS las filas de la tabla izquierda, aunque no tengan coincidencia en la derecha (rellena con NULL).

7.3 GROUP BY y funciones de agregacion

-- Cuantos pedidos tiene cada usuario
SELECT user_id, COUNT(*) as total_orders, SUM(total) as total_spent
FROM orders
GROUP BY user_id
HAVING COUNT(*) > 5;  -- solo los que tienen mas de 5 pedidos

7.4 Indices

Pregunta: "Que es un indice y cuando lo usas?"
Un indice es como el indice de un libro: permite encontrar datos rapido sin recorrer toda la tabla. Crealos en columnas que uses frecuentemente en WHERE, JOIN o ORDER BY. Desventaja: ocupan espacio y hacen los INSERT/UPDATE mas lentos.
CREATE INDEX idx_users_email ON users(email);

7.5 Transacciones

BEGIN;
  UPDATE accounts SET balance = balance - 100 WHERE id = 1;
  UPDATE accounts SET balance = balance + 100 WHERE id = 2;
COMMIT;  -- si todo OK, guarda los cambios
-- Si algo falla: ROLLBACK; deshace todo
Explicacion: Las transacciones garantizan que un grupo de operaciones se ejecuten TODAS o NINGUNA. Si transfieres dinero, no quieres que se reste de una cuenta pero no se sume a la otra.

8.2 Dockerfile basico

# Directorio de trabajo dentro del contenedor

9.3 Redes - Conceptos clave

Pregunta: "Que pasa cuando escribes una URL en el navegador?"
1. El navegador busca la IP del dominio via DNS
2. Establece conexion TCP con el servidor (3-way handshake)
3. Si es HTTPS, negocia TLS/SSL (certificado, encriptacion)
4. Envia peticion HTTP (GET / HTTP/1.1)
5. El servidor procesa y devuelve la respuesta (HTML, JSON...)
6. El navegador renderiza el contenido (parsea HTML, aplica CSS, ejecuta JS)

Pregunta: "Que es CI/CD?"

Pregunta: "Como funciona HTTPS?"

14.4 Conceptos clave para la certificacion

Regiones y Availability Zones
AWS tiene regiones en todo el mundo (eu-west-1 = Irlanda, eu-south-2 = Espana). Cada region tiene multiples Availability Zones (data centers independientes). Despliegas en multiples AZs para alta disponibilidad: si un data center se cae, los otros siguen funcionando.
Modelo de responsabilidad compartida
AWS se encarga de: seguridad fisica, hardware, red global, hypervisor.
Tu te encargas de: tu SO, tu app, tus datos, configuracion de seguridad (security groups, IAM), encriptacion.
Regla facil: AWS protege LA nube. Tu proteges lo que hay EN la nube.
Modelo de precios
- On-Demand: pagas por hora/segundo. Sin compromiso. Mas caro.
- Reserved: compromiso 1-3 anos. 30-70% descuento.
- Spot: capacidad sobrante de AWS. Hasta 90% descuento pero te pueden quitar la instancia.
- Free Tier: 12 meses gratis con limites (750h EC2 t2.micro, 5GB S3, etc.)

15. Terraform - Infraestructura como Codigo

Terraform permite definir infraestructura en archivos de texto y crearla automaticamente. Es como pasar de Docker manual a Docker Compose, pero para TODA tu infraestructura.

15.1 Que es Infrastructure as Code (IaC)

Pregunta: "Que es Infrastructure as Code?"
En vez de crear servidores, redes y bases de datos haciendo clic en un panel web, los defines en archivos de codigo. Ventajas:
- Reproducible: puedes crear el mismo entorno 100 veces de forma identica
- Versionado: los archivos van en Git, ves quien cambio que y cuando
- Automatizable: un pipeline de CI/CD puede crear o destruir infraestructura
- Documentacion viva: el codigo ES la documentacion de tu infraestructura

15.2 Como funciona Terraform

Pregunta: "Explicame el flujo de Terraform"
1. Write: Escribes archivos .tf describiendo la infraestructura que quieres
2. Plan: terraform plan muestra QUE va a crear/cambiar/destruir (sin hacer nada todavia)
3. Apply: terraform apply ejecuta los cambios en el proveedor cloud
4. State: Terraform guarda el estado actual en un archivo terraform.tfstate para saber que existe y que no

15.3 Sintaxis basica (HCL)

# Decir que proveedor usamos
provider "aws" {
  region = "eu-west-1"    # Irlanda
}

# Crear un servidor EC2
resource "aws_instance" "mi_servidor" {
  ami           = "ami-0c55b159cbfafe1f0"  # Ubuntu
  instance_type = "t3.micro"                # Tipo (2 vCPU, 1GB RAM)

  tags = {
    Name = "servidor-web"
  }
}

# Crear un bucket S3 para archivos
resource "aws_s3_bucket" "backups" {
  bucket = "mi-empresa-backups"
}

# Crear una base de datos RDS
resource "aws_db_instance" "postgres" {
  engine         = "postgres"
  engine_version = "16"
  instance_class = "db.t3.micro"
  db_name        = "miapp"
  username       = "admin"
  password       = var.db_password    # variable, nunca hardcodear passwords
}
Analogia con Docker:
- Dockerfile = define como construir una imagen → main.tf = define que infraestructura crear
- docker build = construye la imagen → terraform plan = muestra que va a hacer
- docker run = ejecuta el contenedor → terraform apply = crea la infraestructura
- docker-compose.yml = multiples servicios → main.tf = multiples recursos

15.4 Variables y Outputs

# variables.tf - Definir variables
variable "environment" {
  description = "Entorno (dev, staging, prod)"
  type        = string
  default     = "dev"
}

variable "db_password" {
  description = "Password de la base de datos"
  type        = string
  sensitive   = true    # no se muestra en logs
}

# Usar variables en recursos
resource "aws_instance" "server" {
  instance_type = var.environment == "prod" ? "t3.medium" : "t3.micro"

  tags = {
    Environment = var.environment
  }
}

# outputs.tf - Mostrar informacion despues de crear
output "server_ip" {
  value = aws_instance.server.public_ip
}

output "db_endpoint" {
  value = aws_db_instance.postgres.endpoint
}

15.6 State (Estado)

Pregunta: "Que es el state en Terraform?"
El archivo terraform.tfstate es donde Terraform guarda que recursos ha creado y su estado actual. Sin este archivo, Terraform no sabe que existe y intentaria crear todo de nuevo.

En equipo: El state se guarda en un backend remoto (S3 + DynamoDB) para que todos los miembros del equipo trabajen con el mismo estado. NUNCA se sube a Git.

Con saber explicar el flujo (write → plan → apply) y los conceptos (state, providers, modules) es suficiente para Junior DevOps.
